Connect your AWS VPC to MongoDB Atlas using VPC Peering for private, low-latency database access ‚Äî and learn when peering is the right choice and when to use alternatives.

## Prerequisites

    - A MongoDB Atlas account with a cluster at tier \*\*M10 or higher\*\*
- An AWS account with a VPC where your workloads run
- \*\*Project Owner\*\* role in MongoDB Atlas
- DNS hostnames and DNS resolution enabled on your AWS VPC

## Let's Start

### When to Use VPC Peering

VPC Peering is the right choice when:

    - Your workloads run inside an AWS VPC in the \*\*same region\*\* as your Atlas cluster
- You need \*\*private connectivity\*\* ‚Äî traffic never leaves the AWS backbone
- You're using Atlas clusters at tier \*\*M10 or higher\*\*
- Your resources run on EKS, EC2, or any service inside the VPC

VPC Peering is \*\*not available\*\* in these cases:

    - Cluster tiers \*\*M0, M2, M5, or Serverless\*\* ‚Äî these don't support peering
- \*\*Multi-cloud peering\*\* ‚Äî Atlas doesn't support peering between different providers (e.g. GCP <--> AWS)
- \*\*Overlapping CIDRs\*\* ‚Äî if your AWS VPC CIDR overlaps with Atlas's CIDR, peering can't be established

:::warning
If any of the above applies to you, skip to [Alternative: Whitelist your NAT Gateway IP](#alternative-whitelist-your-nat-gateway-ip).
:::

### Alternative: Whitelist your NAT Gateway IP

If peering isn't an option, you can allow Atlas connections from your VPC by whitelisting the public IP of your \*\*NAT Gateway\*\*:

    1. In the AWS Console, go to \*\*VPC ‚Üí NAT Gateways\*\* and copy the \*\*Elastic IP\*\* associated with your NAT Gateway.
1. In MongoDB Atlas, go to \*\*Network Access ‚Üí IP Access List ‚Üí Add IP Address\*\*.
2. Enter the Elastic IP with a description like \`NAT Gateway - production\`.

Your workloads connect to Atlas via the internet through the NAT ‚Äî no peering required.

:::info
This approach has higher latency than peering but works with any cluster tier and requires no CIDR planning.
:::

### Step 1 ‚Äî Gather Your AWS Information

You'll need the following values from your AWS account before configuring the peering connection in Atlas.

\*\*Get your Account ID\*\*

Click your username in the top-right corner of the AWS Console. Your 12-digit \*\*Account ID\*\* appears in the dropdown.

{/\* TODO: screenshot - AWS console top-right dropdown showing Account ID \*/}

\*\*Get your VPC ID and CIDR\*\*

In the AWS Console, search for \*\*VPC\*\* and open \*\*Your VPCs\*\*. Select the VPC where your workloads run and note the \*\*VPC ID\*\* (starts with \`vpc-\`) and the \*\*IPv4 CIDR\*\* (e.g. \`10.0.0.0/16\`).

{/\* TODO: screenshot - VPC list showing VPC ID and IPv4 CIDR columns \*/}

| \*\*Field\*\* | \*\*Where to find it\*\* |
| --- | --- |
| \*\*Account ID\*\* | Top-right dropdown in the AWS Console |
| \*\*VPC ID\*\* | VPC ‚Üí Your VPCs (starts with \`vpc-\`) |
| \*\*VPC CIDR\*\* | VPC ‚Üí Your VPCs ‚Üí IPv4 CIDR column |
| \*\*Application VPC Region\*\* | The AWS region where your VPC lives |

### Step 2 ‚Äî Create the Peering Connection in Atlas

    1. In MongoDB Atlas, go to \*\*Network Access\*\* in the left sidebar.
1. Click the \*\*Peering\*\* tab, then \*\*Add Peering Connection\*\*.

{/\* TODO: screenshot - Atlas Network Access ‚Üí Peering tab \*/}

    3. Select \*\*AWS\*\* as the cloud provider and fill in the form with the values from Step 1:

    - \*\*Account ID\*\* ‚Äî your 12-digit AWS account number
    - \*\*VPC ID\*\* ‚Äî the VPC ID starting with \`vpc-\`
    - \*\*VPC CIDR\*\* ‚Äî your VPC's IP range
    - \*\*Application VPC Region\*\* ‚Äî the AWS region
    - \*\*Atlas VPC CIDR\*\* ‚Äî Atlas will suggest a range (e.g. \`192.168.240.0/21\`); keep it unless it conflicts with your network

{/\* TODO: screenshot - Atlas Peering connection form filled in \*/}

    4. Click \*\*Initiate Peering\*\*.

Atlas will show the connection with status \*\*"Waiting for Approval"\*\*. The request must now be accepted from the AWS Console.

{/\* TODO: screenshot - Atlas Peering list showing "Waiting for Approval" status \*/}
    ---

### Step 3 ‚Äî Accept the Peering Request in AWS

    1. In the AWS Console, go to \*\*VPC ‚Üí Peering connections\*\*.
1. You'll see a connection with status \*\*"Pending acceptance"\*\*.

{/\* TODO: screenshot - AWS VPC Peering connections list with Pending acceptance status \*/}

    3. Select the connection, click \*\*Actions ‚Üí Accept request\*\*.
3. Review the details in the modal (VPCs, CIDRs, regions) and confirm.

{/\* TODO: screenshot - AWS Accept request modal \*/}

Once accepted, the connection status changes to \*\*"Active"\*\* in AWS and \*\*"Available"\*\* in Atlas.

{/\* TODO: screenshot - Atlas Peering list showing "Available" status \*/}
    ---

### Step 4 ‚Äî Configure Route Tables

For traffic to flow through the peering connection, you need to add a route to the Route Table of the subnets where your workloads run.

    1. In the AWS Console, go to \*\*VPC ‚Üí Route Tables\*\*.
1. Select the Route Table associated with your workload subnets.
2. Open the \*\*Routes\*\* tab and click \*\*Edit routes ‚Üí Add route\*\*:

    - \*\*Destination\*\*: the Atlas VPC CIDR (e.g. \`192.168.240.0/21\`)
    - \*\*Target\*\*: select \*\*Peering Connection\*\* and choose the connection you just created

4. Click \*\*Save changes\*\*.

{/\* TODO: screenshot - AWS Route Tables Edit routes dialog with Atlas CIDR and Peering target \*/}
    ---

### Step 5 ‚Äî Configure a Security Group

Create a Security Group that allows outbound traffic to MongoDB Atlas ports:

    1. In the AWS Console, go to \*\*EC2 ‚Üí Security Groups ‚Üí Create security group\*\*.
1. Add an \*\*outbound rule\*\*:

| \*\*Type\*\* | \*\*Protocol\*\* | \*\*Port range\*\* | \*\*Destination\*\* |
| --- | --- | --- | --- |
| Custom TCP | TCP | 27015 ‚Äì 27017 | Atlas VPC CIDR (e.g. \`192.168.240.0/21\`) |

    3. Attach this Security Group to the EC2 instances, EKS node groups, or other resources that need to connect to Atlas.

:::tip
If you're using \*\*EKS with Karpenter\*\*, add the following tag to the Security Group so Karpenter discovers and applies it automatically:

| \*\*Key\*\* | \*\*Value\*\* |
| --- | --- |
| \`karpenter.sh/discovery\` | 'cluster-name' |

:::
    ---

### Step 6 ‚Äî Add Your VPC CIDR to the IP Access List

Creating the peering connection alone is not enough ‚Äî you also need to allow access at the network level.

In MongoDB Atlas, go to \*\*Network Access ‚Üí IP Access List ‚Üí Add IP Address\*\* and enter your \*\*AWS VPC CIDR\*\* (e.g. \`10.0.0.0/16\`).

{/\* TODO: screenshot - Atlas IP Access List with VPC CIDR added \*/}

### Step 7 (Optional) ‚Äî Connect Through a VPN

If your team accesses MongoDB Atlas through a VPN like Pritunl, standard DNS resolution won't work over peering by default. Follow these steps to enable it:

\*\*Enable Custom DNS in Atlas\*\*

In Atlas, go to \*\*Project ‚Üí Settings\*\* and enable \*\*"Using Custom DNS on AWS with VPC peering"\*\*.

{/\* TODO: screenshot - Atlas Project Settings with Custom DNS option enabled \*/}

\*\*Use the Private Peering connection string\*\*

When connecting to your cluster, select \*\*"Private IP for Peering"\*\* in the connection modal. This provides a hostname that resolves to the cluster's private IP.

{/\* TODO: screenshot - Atlas connection modal with "Private IP for Peering" selected \*/}

\*\*Add the Atlas CIDR route to Pritunl\*\*

In your Pritunl server settings, add the Atlas VPC CIDR as an additional route so VPN clients can reach it.

{/\* TODO: screenshot - Pritunl route configuration with Atlas CIDR \*/}

Connect to the VPN and test the connection using the new private address.

## Verification

\*\*From a private EC2 instance:\*\*

\`\`\`bash
dnf install -qy mongodb-mongosh-shared-openssl3
mongosh "mongodb+srv://your-cluster.mongodb.net/" --apiVersion 1 --username your-user
\`\`\`

\*\*From a Pod in EKS:\*\*

\`\`\`yaml
apiVersion: v1
kind: Pod
metadata:
  name: mongodb-client
spec:
  containers:

        - name: mongodb-client
      image: mongo:latest
      command: ["/bin/sh", "-c", "sleep 3600"]
\`\`\`

\`\`\`bash
kubectl exec -it mongodb-client -- 
  mongosh "mongodb+srv://your-cluster.mongodb.net/" --apiVersion 1 --username your-user
\`\`\`

A successful connection confirms peering is working correctly.
